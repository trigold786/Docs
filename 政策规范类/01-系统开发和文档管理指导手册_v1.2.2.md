## 第一部分：强制阅读&必须执行（30分钟读完，所有条款100%遵守）
### 1.1 开发L1铁则（违反即无效，无豁免）
1. 架构分层禁止逆向调用，依赖层级不超过3层
2. 端口统一使用30000-50000段，非整百端口给开发/测试，整百端口给生产前端，一一绑定
3. 所有IP/端口/计算资源先登记后使用，单服务CPU≤4核、内存≤8G
4. 安全基线必须达标：参数化查询防注入、输出转义防XSS、接口必带幂等键
5. 接口规范：RESTful格式、路径带版本号、统一返回格式、限流响应头必填
6. 事件通信Kafka副本≥3，消费者必须幂等，失败消息进死信队列
7. 版本号用语义化规则，重大升级必须提供迁移文档
8. 模块/仓库/Commit命名必须符合统一规范
9. 单元测试覆盖率达标、PR必须1人以上审批、安全扫描无高危漏洞
10. 调试必须先复现定位根因，未复现禁止提交修复方案
11. 任务拆分规则：所有开发任务必须拆分到30分钟以内可完成粒度，每个任务明确「修改路径+代码范围+验证步骤+验收标准」
12. 子agent开发规则：每个子agent任务必须过两轮强制评审：第一轮评审是否符合需求规范，第二轮评审代码质量/测试覆盖率/安全问题
13. TDD强制规则：所有功能开发必须执行红黄绿流程：先写失败测试用例→确认测试失败→写最小代码让测试通过→重构优化，禁止先写代码后补测试
14. Git隔离规则：每个开发任务必须创建独立工作树分支，禁止在主分支直接开发，任务完成后销毁工作树
15. 验证优先规则：所有功能上线必须提供可量化验证证据（测试通过率/性能指标/验证结果），禁止无验证宣称功能可用

#### 1.1.1 统一命名规范（强制遵守）
##### （1）产品命名规则（对外用户侧）
- 企业可定义统一产品前缀，同系列产品使用统一前缀，例如「Nuero」
- 对外产品名与内部开发代号必须隔离，所有用户侧场景仅使用产品名，禁止暴露内部代号
- 举例：中小微商家SaaS产品对外名称可定为NueroMiniBizSuite，简称NMBS，面向用户的所有场景使用该名称
##### （2）开发项目代号规则（对内内部使用）
- 开发代号可自定义格式，建议规则简洁、可扩展，无歧义
- 举例：可采用「自定义前缀+3位数字」格式，例如前缀为winning时，winning001可缩写为w001
- 所有内部文档、代码、任务安排统一使用开发代号，无需对外透露

### 1.2 架构&选型强制要求
1. 优先复用已有基础组件，禁止重复造轮子
2. 技术栈统一：后端Java/Go、前端Vue/React
3. 数据库强制使用阿里云RDS PostgreSQL，禁用MySQL
4. 中间件仅用官方稳定版，禁止测试版/RC版上线
5. 新产品必须对齐已有架构路线，禁止私自选型
6. 架构变更必须提交ADR文档，审批后才可执行

### 1.3 环境特殊规则（降低资源占用专用）
1. 生产环境必须严格执行高可用要求：中间件副本≥3、服务多实例部署
2. 研发/测试/UAT非生产环境可特殊处理：
   - 中间件可使用单副本，省略高可用配置
   - 服务可单实例部署，无需冗余
3. 所有测试环境必须明确标识，禁止接入生产流量、存储生产敏感数据

### 1.4 文档管理强制规则
#### 1.4.1 目录结构（通用标准）
```
[项目总根目录]/
├── 00-docs/          # 统一文档根目录
│   ├── docs-tech/    # 技术类文档
│   └── policies-guidelines/ # 政策规范类
│       ├── policy/   # L1核心制度（本文档所在目录）
│       ├── regulation/ # L2强制规定
│       └── archived-documentation/ # 归档过时文档
├── 01-dev-project/   # 开发项目根目录
├── 02-subagent-project/ # 子代理项目
├── 03-uiux-project/  # UI/UX项目
└── project_records/  # 全量项目记录归档目录
```
- 根目录散落文档必须3天内归类，禁止长期存放
- 所有项目记录必须统一归档到总目录下的project_records目录，禁止单独存放

#### 1.4.2 记录留存强制规则
1. 所有项目产出必须永久留存，无遗漏：包含会议纪要、所有版本文档、代码提交记录、评审记录、测试报告、上线日志、决策记录、运营数据等
2. 所有留存记录必须可追溯，明确修改人、修改时间、修改原因
3. 归档文件禁止删除、篡改，仅可新增，历史版本永久留存

#### 1.4.3 命名规则
1. 顶级目录建议用数字前缀分类：00=文档类、01=开发类、02=测试类、03=UI类，可根据项目调整
2. 版本号后缀：`文件名vX.X.X.md`，文件名版本必须和文件内版本一致
3. 过时文档命名：`过时-原文件名.md.bak.YYYYMMDD`，归档到对应目录
4. 禁止使用无意义文件名，必须见名知意

#### 1.4.4 版本管理
- 版本规则：v主版本.次版本.补丁号
  - 主版本+1：核心结构/规则重大变更
  - 次版本+1：新增内容/规则
  - 补丁号+1：修正错误/格式
- 所有正式文档必须带版本头部，明确版本、更新时间、更新说明

### 2.1 文档分类参考
| 分类 | 级别 | 用途 |
|------|------|------|
| policy | L1 | 核心制度，最高权威 |
| regulation | L2 | 强制规定，违反有后果 |
| guideline | L3 | 指导性建议 |
| standard | L4 | 公开/内部标准 |
| instruction | L5 | 操作手册/接口规范 |
| reference | L6 | 参考资料 |

### 2.2 常用工具指南
1. 端口占用检查：`lsof -i :端口号`
2. 文档编码检查：`file -i 文件名`
3. 归档命令：`mv 原文件名 过时-原文件名.md.bak.$(date +%Y%m%d)`
